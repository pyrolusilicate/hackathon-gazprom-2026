# Парсинг PDF в Markdown формат | Хакатон

## Задача
---
Данный проект представляет собой пайплайн для преобразования разрозненных PDF-документов в структурированный формат Markdown. 

Основная цель — максимально точное извлечение текста, таблиц (включая объединённые ячейки и многоуровневые заголовки), изображений и списков с сохранением оригинальной иерархии документа (reading order). Решение спроектировано для работы в условиях ограниченных вычислительных ресурсов (до 1 GPU, <40 Гб VRAM) с использованием исключительно открытых (open-source) локальных моделей.

Особое внимание уделено фильтрации "мусора" (водяные знаки, колонтитулы, номера страниц), обработке растрированных сканов и корректному сохранению изображений строго в соответствии с требованиями проверяющей системы.

---

## Архитектура решения

Решение построено на гибридной 3-ступенчатой архитектуре, которая комбинирует скорость классических парсеров с мощностью современных визуальных LLM для сложных случаев:

### 1. Router (Layout Analysis)
В качестве маршрутизатора используется модель **DocLayout-YOLOv10** (`layout_router.py`).
* **Multi-scale inference:** Используется ансамбль разрешений (1280 и 2400) для точного захвата широких таблиц и разбиения сросшихся элементов.
* **Reading Order:** Детерминированный алгоритм выстраивает порядок чтения (логические блоки -> горизонтальные полосы -> колонки слева направо -> сверху вниз).
* **Умная фильтрация:** На этом этапе отбрасываются элементы вроде `page-header`, `page-footer`, `watermark`, чтобы они не попали в итоговый Markdown.

### 2. Fast Track (Основной движок)
Для чистого извлечения текстов и таблиц используется **Docling (IBM)** (`docling_engine.py`).
* Выполняет быстрое извлечение текстовых слоев и структуры таблиц (используется TableFormer).
* Блоки Docling сопоставляются с Bounding Boxes от YOLO через метрику **IoM**.
* Автоматически обрабатывает объединенные ячейки таблиц, выполняя forward-fill слева направо (`_forward_fill`).

### 3. Fallback (Сложные кейсы и сканы)
В случаях, когда Fast Track не справляется (например, документ отсканирован без текстового слоя, таблица отрендерена как картинка или YOLO обнаружил фигуру с текстом), используется тяжелая модель — **olmOCR-2-7B-1025** на базе Qwen2.5-VL (`olm_engine.py`).
* Изображение обрезается по YOLO-координатам (`crop_pdf_bbox`) и передается в VLM.
* Отрабатывает детерминированно (`temperature=0.1`, `do_sample=False`) в BF16.
* Встроенные анти-галлюцинационные валидаторы (`cyrillic_ratio`, `repetition_ratio`) предотвращают попадание сгенерированного мусора в финал.

## Структура репозитория

```text
├───data
│   ├───output                   # Сгенерированные .md файлы и финальный submission.zip
│   │   ├───images               # Сохраненные картинки в формате doc_<id>_image_<N>.png
│   │   └───temp                 # Временные файлы
│   ├───raw                      # Исходные файлы document_NNN.pdf
│   └───visualization            # Дебаг-рендер YOLO боксов (при запуске с --visualize)
├───src
│   ├── config.py                # Центральные константы (DPI, пути, пороги IoM)
│   ├── content_extractor.py     # Валидаторы, форматирование Markdown и анти-галлюцинации
│   ├── coord_projection.py      # Трансляция координат между PDF-points и пикселями YOLO
│   ├── device.py                # Выбор устройства (CUDA/MPS/CPU) и аллокация VRAM
│   ├── docling_engine.py        # Обёртка для Fast Track (Docling DocumentConverter)
│   ├── layout_router.py         # YOLOv10 роутер и определение порядка чтения (reading order)
│   ├── olm_engine.py            # Обёртка для Fallback (olmOCR / Qwen2.5-VL)
│   └── pipeline.py              # Главный оркестратор пайплайна (сборка, склейка таблиц, zip)
└───weights                      # Директория для локального кэширования весов YOLO
```

## Инструкция по развертыванию и запуску

Для управления зависимостями и виртуальным окружением используется `uv`.

**1. Установка зависимостей:**
В корневой папке проекта выполните команду для синхронизации окружения:
```bash
uv sync
```

**2. Подготовка данных:**
Поместите все исходные PDF-файлы (вида `document_NNN.pdf`) в папку `data/raw/`.

**3. Использование (CLI):**
Пайплайн запускается через скрипт `pipeline.py`. Доступны следующие сценарии запуска:

Запустить обработку всех PDF-файлов в папке `data/raw` (сгенерирует `submission.zip`):
```bash
uv run src/pipeline.py --all
```

Запустить обработку только одного конкретного документа (удобно для тестов):
```bash
uv run src/pipeline.py --pdf data/raw/document_001.pdf
```

Запустить массовую обработку **только** через Fast Track (отключить olmOCR для экономии VRAM / ускорения на слабых машинах):
```bash
uv run src/pipeline.py --all --no-vlm
```

Запустить распознование объектов только одного конкретного документа (--visualize сохраняет их фото в папку data/visualize):
```bash
uv run src/layout_router.py --pdf data/raw/document_001.pdf --visualize
```

**Результат:** Итоговые Markdown-файлы и папка `images` будут сформированы в директории `data/output`. При использовании флага `--all` скрипт автоматически запакует их в `submission.zip`.
